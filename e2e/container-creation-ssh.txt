# Container Creation and SSH Access Test

## Instructions

Test container creation, SSH access, and basic container operations.

### Prerequisites

- Registration must be completed (setup-and-basics.txt)
- exed and sshpiper must be running
- SSH key must be created

### Test Steps

#### Container Creation and SSH Access

4. Create a new container using "ssh -p 2222 localhost"

5. SSH into that container (by using ssh -p 2222 name@localhost) and run "id"

6. Open up a shell to the new container and check the hostname.

7. Use scp with the new container to test that SSH subsystems work.

## Edge Cases

11. Creating new machines should have restrictions on the names of new machines.
For example, "--foo" is not a valid machine name. Make sure that returns an error message.

12. If the user doesn't have an SSH key (which can be simulated with
 "ssh -o PreferredAuthentications=password -o PubkeyAuthentication=no -p 2222 localhost"
they should get an error message indicating how to create a key.


## Hints

### Container Creation Hints
- Type `new` at exe.dev prompt. Creates machine with random name like "able-yankee". Shows command like "ssh -p 2222 able-yankee@localhost"

### SSH Access Hints
- Use `ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null -p 2222 machinename@localhost`. Wait ~10-15 seconds after container creation for SSH daemon to be fully ready. If connection fails, check sshpiper logs with `tmux capture-pane -p -t testing:sshpiper` for connectivity issues.
- In container shell, run `hostname`. Should show "machine-name.team-name.exe.dev"
- Use `scp -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null -P 2222 localfile machinename@localhost:/remote/path`. Ensure SSH connection works first before testing SCP.
