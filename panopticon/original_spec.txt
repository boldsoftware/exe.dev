Design spec for Panopticon. It will live on an EXE VM at panopticon.exe.xyz.

Commander's intent: a dSPY RLM-based platform for absorbing all available information about the EXE world. Users' comments on Discord, inbound support emails, public GitHub issues, private GitHub issues, the EXE code and git history, our internal Slack, the stream of logs via Honeycomb, all of it. Multiple agents built on the same platform consume this data for different purposes: a newsletter agent synthesizes daily activity summaries, an issue gardener curates a prioritized bug/feature list, a CRM monitor tracks user interactions. The platform provides the data absorption layer; agents provide the intelligence. Humans are in the loop as a security barrier -- if something untoward slips through via prompt injection, it can't cause harm.

This is not a standard coding agent because many data sources are untrusted, which means we cannot just give it tools. We use dSPY (lowercase d, capital SPY) and RLM (capital RLM) -- a system for exposing information to an agent and letting it manipulate that information in Python, recursively calling sub-agents as needed.

Out-of-the-box RLM would require serializing all external data sources to strings upfront. That's wrong for three reasons: too much eager work (most of it unused), total string size too large, and not discoverable for agents. Instead, we give the agent Python objects with useful methods and attributes that resolve to strings lazily.

RLM does not natively support passing arbitrary Python objects -- its `_serialize_value()` only handles JSON primitives. We will extend the RLM library: modify `_serialize_value()` to handle our proxy objects, and add a tool-call bridge so lazy attribute access can punch through to the host. We pass into RLM one string explaining: here are your Python objects, inspect them via docstrings, manipulate them. Progressive disclosure -- the agent inspects what's available, drills into attributes, lazily loads data as needed.

The top-level Python objects represent data sources. A Discord object has `.channels`, which expands to channel objects with their own attributes. Everything is docstringed, uses progressive disclosure, and is manipulable in Python. The agent navigates these to find its way through the world. Methods expose remote capabilities (search, server-side filtering, timestamp conversion) that can't be replicated or are more efficient than filtering cached data locally. The API trusts the agent to decide how best to use these capabilities -- the constraint is security, not workflow. See proxy_api_design.md for the full design philosophy.

Most objects are read-only. The agent cannot write to Discord or GitHub. Writable surfaces are limited and agent-specific: the issue gardener writes to a git-backed issue set; the newsletter agent writes to an output field. The entire interaction with the external world is mediated by these Python objects.

dSPY/RLM agents run inside a Deno sandbox so they can't spawn subprocesses or exec arbitrary code. For the lazy proxy objects to fetch data, we poke exactly one hole in the sandbox: a single UNIX socket, granted via Deno's path-scoped `--allow-read` and `--allow-write` on that socket path. All data source proxies mux through this one socket.

The data source architecture is pluggable. Adding a new source means: define the Python objects, define a handler on the host side of the mux, slot it in.

On the sandbox side, proxy objects make JSON-over-HTTP requests through the UNIX socket. On the host side, a multiplexing proxy server routes requests to the appropriate data source handler. The transport is JSON over HTTP -- not pickle, which is unsafe. Each handler is strongly typed and safe by construction: "fetch issue #N from the exe.dev repository" is a valid request; "fetch arbitrary URL" is not. Safety by construction means every exposed operation must be safe for ALL possible inputs against a trusted service. We trust GitHub, Discord, etc. as services -- search queries, pagination params, and other agent-controlled inputs are safe because they go only to services we trust and produce only read effects. There cannot be an API that lets you post to GitHub or fetch an arbitrary resource, because those could exfiltrate data or produce write effects visible to untrusted third parties. See proxy_api_design.md for the full safety model.

Key design principle: all lazily loaded proxy objects work through the mux intermediary. The intermediary is a rigid straightjacket enforcing safe interactions with the external world. The worst a rogue agent can do is trash its own writable surface (e.g. the issue set), and that's recoverable because everything is copy-on-write. Discover the damage, roll back to a known-good state, track down the bad data source, delete it, resume. Part of the agent's prompting includes watching for sketchy content and flagging it.

The issue gardener is one agent built on this platform. Piece one of the gardener: a Panopticon web server, where users see a prioritized list of issues and interact with them. Human users have privileged rights: modify, delete, adjust priority, lock an issue. Anything a human does is associated with the issue in an unalterable manner, so agents must respect human actions. Each issue has one or more labels and a set of priority scores: ease of implementation, risk, urgency, frequency, user pain. Beyond metadata, each issue carries a list of user/agent actions and freeform Markdown-ish content rendered as plain text. The gardener agent runs in a loop, waking up periodically to comb through all data sources and garden the issue set: adding, removing, improving, re-prioritizing.

Storage for the issue gardener is a git repository. Each issue is a file (frontmatter + markdown body), each mutation is a commit. Git provides copy-on-write by definition -- every commit is an immutable snapshot. Rollback is `git checkout <sha>`. At hundreds of issues with mutations every few minutes, git performance is more than sufficient. The data is human-inspectable with any text editor. Agent notes live in a `notes/` directory in the same git repo -- slug-keyed text files with the same COW semantics.

The host-side proxy handlers cache aggressively -- the system operates on agent time scale, not human time scale. Cache external data for up to an hour. Don't cache through writes (the agent would get confused). Docstrings note that data is aggressively cached, so the agent expects stale cross-references and doesn't spin its wheels. This is reinforced at the system prompt level.

To recap: Panopticon is a platform for absorbing untrusted data sources through a single muxed UNIX socket into Deno-sandboxed dSPY/RLM agents. Data sources are lazily loaded Python proxy objects with rich docstrings and agent-empowering methods. Agents built on this platform -- newsletter, issue gardener, CRM, and more -- consume the same data sources for different purposes, each with its own output surface and cadence.
