To avoid more product cert issues, I set up an AWS staging env for exed.

It was entirely focused on testing certs. It re-used the existing ctrhost (danger!) and ignored everything not needed for certs.

This is a very raw, _extremely_ lightly edited transcript of a conversation with Codex in which it walked me through setting it up, complete with false steps and mistakes and confusions. Attempts to get an LLM to clean it up and organize it have utterly failed, and I don't want to put in the time now. I'm leaving it here as a maybe-useful reference, but definitely _not_ a how-to guide.

### Codex transcript

  AWS Prep

  - IAM role: Create a dedicated role (e.g. exed-staging-role) with the EC2 trust policy and attach an inline policy that allows route53:ListHostedZones, route53:ListHostedZonesByName, route53:ListResourceRecordSets, and
    route53:ChangeResourceRecordSets scoped to the exe-staging.dev hosted zone ARN plus route53:GetHostedZone. Attach the role to an instance profile so EC2 can assume it.
  - Security group: Allow inbound TCP 22 (SSH admin), 80 (plain HTTP for redirects/ACME), 443 (HTTPS), and optionally 2223 if you want to test ssh-on-exed. Outbound can stay “all traffic”.
  - Elastic IP: Allocate one up front; you’ll need a stable IPv4 address before you create the A / wildcard records in Route53.
  - Instance: Launch Ubuntu 24.04 LTS (same AMI family ops/setup-host-part1.sh uses) in us-west-2, type t3.large or bigger so Go’s TLS jobs aren’t CPU starved. 30 GB gp3 root volume is plenty for the SQLite DB plus cached certs. Attach
    the security group, IAM instance profile, and the Elastic IP.

  ———

  Instance bring-up

  1. SSH in once it’s reachable: ssh ubuntu@<elastic-ip>.
  2. Update packages and install basics you’ll want for debugging:

     sudo apt-get update
     sudo apt-get install -y sqlite3 jq unzip
     sudo mkdir -p /home/ubuntu/certs
     sudo chown ubuntu:ubuntu /home/ubuntu/certs
  3. Optional but nice: install tailscale if you want a backdoor even when security groups change.

  ———

  Build + ship exed

  4. On your dev machine, build the Linux binary exactly like prod does (ops/deploy-exed.sh already does this):

     GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build -o /tmp/exed-staging ./cmd/exed
     Make sure you include whatever branch-specific flag sets the hostname to exe-staging.dev.
  5. Copy the binary and supporting data up:

     scp /tmp/exed-staging ubuntu@<elastic-ip>:~/exed-staging
     scp ghuser/whoami.sqlite3 ubuntu@<elastic-ip>:~/whoami.sqlite3
  6. On the instance:

     chmod +x ~/exed-staging
     sqlite3 ~/exe-staging.db 'pragma journal_mode=WAL;'
  7. Grant the binary cap_net_bind_service so it can listen on 80/443 without running as root, or let systemd do it:

     sudo setcap 'cap_net_bind_service=+ep' ~/exed-staging

  ———

  Systemd service (recommended)

  8. Drop an env file (/etc/default/exed-staging) similar to prod’s but trimmed for staging:

     HOME=/home/ubuntu
     PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
     GITHUB_TOKEN=<fine-grained token, public repo scope is enough>
     POSTMARK_API_KEY=
  9. Copy ops/exed.service to /etc/systemd/system/exed-staging.service and adjust:
      - Description=exe.dev staging
      - WorkingDirectory=/home/ubuntu
      - EnvironmentFile=/etc/default/exed-staging
      - ExecStart=/home/ubuntu/exed-staging -http=:80 -https=:443 -ssh=:2223 -piper-plugin=:2224 -piperd-port=22 -db=/home/ubuntu/exe-staging.db -gh-whoami /home/ubuntu/whoami.sqlite3 -containerd-addresses "" -hostname exe-staging.dev
  10. Reload + start:

     sudo systemctl daemon-reload
     sudo systemctl enable --now exed-staging
     sudo journalctl -fu exed-staging
  ———

  Route53 records for exe-staging.dev

  11. Apex A record: exe-staging.dev -> <elastic-ip>.
  12. xterm.exe-staging.dev A record to the same IP.
  13. Wildcards (A or ALIAS) for *.exe-staging.dev and *.xterm.exe-staging.dev, both pointing to the Elastic IP. Those ensure boxname.exe-staging.dev and boxname.xterm.exe-staging.dev resolve before certificates are requested.
  14. Set TTLs to something short (60 s) while testing so you can move the IP if needed.
  15. Verify with dig exe-staging.dev +short, dig box1.exe-staging.dev +short, etc., before you hit HTTPS.

  Because the wildcard manager requests both the base and xterm certificates (execore/exe-web.go:95-139), the first TLS handshake to either hostname will trigger two DNS-01 challenges _acme-challenge.exe-staging.dev and _acme-
  challenge.xterm.exe-staging.dev. You can watch them with dig TXT _acme-challenge.exe-staging.dev.

  ———

  TLS verification & monitoring

  - Hit the health endpoint to force certificate issuance:

    curl -vk https://exe-staging.dev/health
    curl -vk https://box1.exe-staging.dev/   # should reuse the wildcard cert
    curl -vk https://box1.xterm.exe-staging.dev/
  - Use openssl s_client -connect exe-staging.dev:443 -servername exe-staging.dev | openssl x509 -noout -text to confirm the SANs include both the apex and wildcard.
  - Tail logs with journalctl -fu exed-staging while certs are provisioning; failures will bubble up from the route53/wildcard_cert.go path.
  - Check the cached cert/key pair lands under /home/ubuntu/certs/ (because autocert.DirCache("certs") uses the working directory).

  ———

  Cleanup / next steps

  1. Once you’ve validated TLS, you can tighten the security group (drop port 80 if you don’t need it, keep 80 only if you want HTTP→HTTPS redirects).
  2. Document any new flags or config files you had to add (e.g., the new hostname flag) so production doesn’t regress.
  3. When you’re done with the staging box, release the Elastic IP and terminate the EC2 instance to avoid lingering IAM roles with Route53 write access.

  Natural follow-ups:

  4. Add a lightweight smoke-test (curl or Go integration) that hits https://exe-staging.dev/health so you can re-run the TLS check after every change.
  5. If you expect to keep this environment around, bake the systemd unit and env file into cloud-init or an AMI so recreating the box is a single ops script run.

› ok, i think i have the iam role and policy set up. security group: where/what do i do here?


• Head to VPC → Security groups in the AWS console:

  1. Click Create security group.
  2. Give it a name like exed-staging-sg, pick the same VPC you’ll launch the EC2 instance in (likely your default VPC unless you’ve set up a dedicated one).
  3. Add inbound rules:
      - Type SSH, Port 22, Source your IP (or 0.0.0.0/0 temporarily if you don’t have a static IP).
      - Type HTTP, Port 80, Source 0.0.0.0/0 (Let’s Encrypt/redirect traffic).
      - Type HTTPS, Port 443, Source 0.0.0.0/0.
      - Optional: Custom TCP, Port 2223, Source your IP if you plan to test the SSH service exed exposes on that port.
  4. Leave the default outbound rule (All traffic, 0.0.0.0/0) so the box can reach Let’s Encrypt and Route53.
  5. Create the group.

  You’ll pick this security group on the EC2 launch wizard later.


› i'm on the launch instance page. i don't see where to assign the elastic ip address i allocated. i only see "Auto-assign public IP" which is a boolean toggle.


• Right—EC2 doesn’t let you attach the Elastic IP during the launch wizard. Leave “Auto-assign public IP” off (or on—it’ll get replaced), finish launching, and once the instance is running:

  1. Go to EC2 → Elastic IPs.
  2. Select the address you allocated.
  3. Click Actions → Associate Elastic IP address.
  4. Choose your new instance (or its primary network interface) and save.

  That swaps whatever temporary public IP the instance had with your Elastic IP.


› ok. how do i ssh in with the pem i downloaded? (what flag to ssh?)


• Use the -i flag to point ssh at the private key file:

  chmod 600 /path/to/exe-staging.pem
  ssh -i /path/to/exe-staging.pem ubuntu@<elastic-ip>

  Replace <elastic-ip> with the address you associated. The default user on Ubuntu images is ubuntu.


› before build + ship, oughtn't i set up A records for exe-staging.dev?

• Yes—set up DNS before you try to hit TLS. Minimal records:

  - exe-staging.dev A → Elastic IP.
  - xterm.exe-staging.dev A → Elastic IP.
  - *.exe-staging.dev A/ALIAS → Elastic IP.
  - *.xterm.exe-staging.dev A/ALIAS → Elastic IP.

  Use Route53 → Hosted zones → exe-staging.dev, add record → “Simple routing”. With those in place the Let’s Encrypt flow can resolve every hostname as soon as exed starts.

  └ Search obtainCertificate in wildcard_cert.go
    Read wildcard_cert.go

• I see that the ACME client is created with proper keys and directory URLs but never calls Register or GetReg to ensure the account is registered with Let's Encrypt. This likely causes the "No Key ID in JWS header" error when placing a
  new order because the client lacks an account ID (KID). The fix involves adding account registration steps in NewWildcardCertManager around lines 117-134 in route53/wildcard_cert.go, either by calling client.Register or client.GetReg to

• That error comes straight from Let’s Encrypt when you send a NewOrder request without an already-registered ACME account. In our wildcard path (route53/wildcard_cert.go:117-134) we construct an acme.Client with a key pulled from cache, but we never call client.Register/client.GetReg to create (or look up) the account before calling AuthorizeOrder. Because NewOrder requires a kid header that points at an existing account, the CA rejects the JWS with “No Key ID in JWS header.”

  So the fix is to register once when NewWildcardCertManager boots—e.g., client.Register(ctx, &acme.Account{TermsAgreed: true}) or GetReg if you’ve already registered and cached the account URL—before you start calling AuthorizeOrder. Once the account exists, Let’s Encrypt assigns it a KID and the client can include that in every subsequent ACME request, which resolves the error.

› OK, but this doesn't happen in prod. So we're storing this somewhere...somewhere that I can presumably manually copy over to staging. Where?

• NewWildcardCertManager points its cache at autocert.DirCache("certs") (route53/wildcard_cert.go:95-134), and the systemd unit that runs prod sets WorkingDirectory=/home/ubuntu (ops/exed.service:15-34). That means prod’s ACME account key
  and cached certs live under /home/ubuntu/certs on the exed-01 box—most importantly the file acme_account+key, plus whatever autocert drops alongside it.

  To reuse the same registered account on staging:
  2. Ensure the directory ownership on staging is ubuntu:ubuntu so the service can read it.
  3. Restart exed on staging so it reloads the cached key (sudo systemctl restart exed-staging).

  With the same cache contents in place, the ACME client will reuse the already-registered Let’s Encrypt account and the “No Key ID in JWS header” error should disappear.
