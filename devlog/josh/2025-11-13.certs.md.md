We had a production outage a week ago around LetsEncrypt wildcard cert issuance. We hit rate limits due to not having a disk cache. We collectively cobbled together enough of a fix to get us live again.

Yesterday I overhauled the code to fix a bunch of bugs. This devlog entry outlines the big picture, to complement the individual commits (all prefixed route53).

Significant fixes:

- Simplify the code by setting up a single fixed list of wildcard domains
	- Currently `exe.dev`, `xterm.exe.dev`, soon to also be `exe.xyz` and `xterm.exe.xyz` (and we'll probably also remove `xterm.exe.dev` once that migration completes)
- Stop attempting to issue wildcard certs for  `*.*.exe.dev` for unknown first level (like xterm)
	- Currently on exed-01, in `~/certs`, `ls -l` shows:
		- `-rw------- 1 ubuntu ubuntu 5294 Nov 11 04:15 'wildcard_cert+*.flying-squirrel.exe.dev'`
- Fix path construction bug causing some disk cache entries not to work
- Singleflight requests to avoid hitting rate limits
- Eagerly fetch on launch
	- Gets a head start on any blocking LE requests we need to make
- Add background refresh near expiry
- Fix acme challenge DNS code that didn't handle subdomains properly
- Group challenges together to avoid paired challenges (wildcard, apex) stomping on each others' work and TXT records

Not fixed (low priority):

- Move persistent cache from disk to db
- Thread server shutdown into the singleflight context for graceful shutdown during LE requests
- Combine requests so we send a single SAN instead of multiple
	- Some trade-offs here, but given that we know the entire list of certs we want, and it is small, we should just ask for everything all at once instead of dribbling it out independently
